# OMEGA Terminal — Design Notes

## Epistemic hierarchy

The pipeline is built around a clear hierarchy:

1. **Reasoning** (commitments, counterarguments, failure conditions, VoI rationale) — generated by the model; treated as the core value. Robust across runs when prompts are stable.
2. **Evidence** (citations, statistics, empirical claims) — either honestly labeled as unverified or computed in code. Treated as lower reliability; never presented as authoritative without verification.
3. **Disagreements** (e.g. `model_order_disagrees_with_computed_voi`) — flagged rather than hidden. When intuitive ranking disagrees with computed VoI, it’s worth human attention (either the model is capturing something the formula misses, or its scores are inconsistent with its reasoning).

This keeps the system more honest than most analyst tools: reasoning is the product; evidence and rankings are explicitly scoped.

---

## Decision engine sources: post-hoc verification over pre-generation RAG

**Preferred path:** Run a **post-hoc verification pass** on the decision engine’s key claims, rather than RAG before generation.

- **Pre-generation RAG** biases the analysis toward whatever retrieval finds, and can dilute the thing that currently works: reasoning from structural principles about incentives and governance instead of pattern-matching to case studies.
- **Post-hoc verification** keeps the model’s reasoning independent, then grounds only the claims that matter: after the decision engine outputs with `source_ref: "general knowledge — not verified"`, a verification step searches for each key claim (e.g. “programme director autonomy blocks infrastructure extraction in research agencies”) and either:
  - upgrades `source_ref` with a real citation when supporting evidence is found, or
  - flags the claim as `"claim unverified — no supporting evidence found in search"`.

That yields honest sourcing without letting retrieval artifacts distort the reasoning. The script supports this via a `--verify-dec` flow and a `verifyDecisionClaims()` stub; plug in your preferred search API (e.g. Tavily, SerpAPI) when ready.

---

## VoI

VoI scores and ranking are computed in code from `impact_1_to_10`, `uncertainty_1_to_10`, `decision_sensitivity_1_to_10`. The model supplies only those three inputs and the prose rationale. Any VoI numbers in the model’s text are ignored; the JSON is authoritative.
